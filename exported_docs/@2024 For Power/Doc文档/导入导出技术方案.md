# 结合现有导入，导出能力分析
## 现状能力分析
1. 现有能力基于 util 工具类的方式
2. 每个导入导出接口都涉及到了 excel 元数据的解析能力
3. 业务导入记录，存在同一张表中



## sdk 提供能力分析，对现有能力的影响
1. 导入服务提供能力描述: 替换原有util 解析excel 的处理
2. 提供灵活性的导入能力配置
3. **<font style="color:#DF2A3F;">让接入导入，导出能力的业务人员只需要关注业务处理，不需要关心具体的excel元数据的基础过滤，对象转化工作</font>**
4. 对导入能力做扩展，不会影响原有的业务能力

# 导入组件设计
## 问题分析
导入数据量过大，导致OOM

原因: 导入的数据过多，全部数据在内存中处理



导入效率慢，灵活性不高 ,异常原因展示不明确

原因: 导入没有分批，导入元数据，过滤规则，过滤结果没有中间存储的信息



图片导入不支持



方向: 

1. 导入使用分批数据
2. 标准化过滤: 全局唯一过滤，字段规则校验，业务规则校验
3. 使用sdk 的方式 + 自定义注解（字段描述，字段过滤） + 中间介质存储 + 注册自定义注入业务处理接口
4. 图片导入，导出 使用 url 方式 或者 resourceId 的方式



## 技术方案
### 依赖模型
| 依赖模型 | | |
| --- | --- | --- |
| 导入记录 **(基于现有的导入模型做扩展)** | 接触配置信息，是否可以中断，<br/>是否异步处理<br/>导入业务码<br/>业务描述<br/>导入进度 。。。。 | 包含基础的导入信息 |
| 导入行 | 业务key，index，是否成功，异常信息，是否可用，是否重复 | 记录行的元数据信息 |

### 使用技术
1. 文件处理工具
    1. EasyExcel （处理批量问题） 缺点: 不支持图片 ，纯文本处理   优点: 批量处理能力强
    2. Apache POI （处理图片文件导入导出） 缺点: 不支持批量的封装  优点: 灵活性高，支持自定义的格式处理，文件处理基于磁盘流
2. 导入元数据存储
    1. mysql 表 （导入记录模型，导入数据行，导入字段信息）
        1. 关键信息，导入进度，导入状态，导入字段元数据，导入字段元数据校验结果，业务校验结果
    2. ~~redis 缓存 （批量分组数据缓存，数据量支持自定义）~~~~** (可选)**~~
        1. ~~ 保存每个批次的处理导入行数据~~
3. 异步能力支持
    1. 线程池
        1. 每次导入请求放入多线程 
        2. 处理批量数据，业务处理
        3. 优点	: 实现简单
        4. 缺点 : 请求顺序无法保证 
    2. mq 
        1. 每次请求导入放入消息队列处理
        2. 处理批量数据，业务处理
        3. 优点: 流量可控，业务处理顺序可控
        4. 缺点: 实现稍微复杂
4. 元数据保存周期
    1. 导入公共组件处理，保存元数据，可设置元数据保存时长
    2. 通过定时任务，基于过期时间 和 处理状态进行删除



### 导入业务核心
**数据过滤:** <font style="color:#DF2A3F;">备注-后期可以通过页面控制</font>

1. 导入excel 数据大多包含唯一性校验 (只能解决局部唯一问题)
    1. 全局唯一: 要么更新，要么丢弃， 根据业务接口处理
    2. 解决途径: 通过中间数据存储，保证局部唯一
    3. 业务场景分析: 	
        1. 如果包含多个重复的数据，选择第一个，第二个，还是最后一个？ 【接入方决定】
2. excel字段校验校验
    1. 通过自定义注解方式
    2. 包含常见的支持: 是否必填， 是否数字，是否必须包含某个字符, 正则校验， 自定义接口回调
    3. 条件过滤组合可控
3. 业务过滤
    1. 具体的业务过滤，通过给定的feign 接口处理

**状态控制(如是否可以中断): **

1. 文件处理状态 (上传中，失败， 处理中，处理成功，失败，处理进度)
2. 业务处理状态 (处理中，处理进度，系统异常)
3. 中断:  无论是否异步都可以中断， 需要提供中断接口， 中断过程中产生的业务影响无法恢复

**多次导入顺序 : **

1. 请求顺序，业务处理数据顺序
2. 需要保证顺序: 使用mq 顺序消息 缺点: (处理能力低下)
3. 不要顺序: 多线程 + mq 都行

**异常数据校验明细: **

1. 下载异常数据，具体的失败原因
2. 目前: 只是显示最新的校验异常
3. 期望： 提供全量展示异常能力

@Retention(RetentionPolicy.RUNTIME)
public static @interface ExcelImporterField {
    // excel 表头
    String title() default "";
    // 字段描述
    String limitDesc() default "";
    // excel 表头
    boolean isRequire() default false;
    // 邮箱
    boolean isEmail() default false;
    // 手机号
    boolean isPhone() default false;
    // 身份证
    boolean isIdCard() default false;
    // url
    boolean isUrl() default false;
    // 包含限制
    String mustContain() default "";
    // 开始限制
    String mustStartWith() default "";
    // 结束限制
    String mustEndWith() default "";
    // 正则
    String regex() default "";
    // 自定义限制（基于属性类型的回调接口）
    Class<? extends CustomLimitCallback> customLimitCallback() default DefaultCustomLimitCallback.class;
    // 是否使用自定义限制
    boolean isUseCustomerLimit() default false;
}

    interface CustomLimitCallback {
        ExcelFilterResult filter(String value);
    }
    public static class DefaultCustomLimitCallback implements CustomLimitCallback{
        @Override
        public ExcelFilterResult filter(String value) {
            return new ExcelFilterResult();
        }
    }
    public static class ExcelFilterResult {
        public Boolean result = true;
        public String failureReason;
    }
