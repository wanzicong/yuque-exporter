| # Outway 任务数据系统<br/>项目描述: <br/>功能: 利用定时任务拉去快手开发平台接口数据，包含但不限于核心业务数据:订单数据，分销单数据，商品数据，售后单数据，团长订单数据，视频数据，评价数据，物流链路数据等。接口分为列表数据，详情数据。拉去数据维度均以主播维度进行拉去。利用这些数据作为报表分析与统计，提供rpc接口服务上游核心业务，如客服系统，物流管理系统等对直播产生的数据强依赖的业务系统。 目前数据体量以达到单表最高20亿+的数据量,拉取任务的机器数量18 台。<br/><br/>项目宏观设计描述: <br/>任务模型采用基于主播生成主任务，定时任务生成子任务。 任务包含了主播信息，数据业务类型，任务状态，最新任务执行时间，子任务步长。 <br/>定时任务设计包含了业务数据拉去，任务状况监控，任务执行频率，分片，状态，参数范围的自动化分配。<br/><br/>系统难点: <br/>数据拉取实时性较高，数据延迟不高于半小时。<br/>数据完整性较高和数据一致性较高。<br/><br/>目前系统问题: <br/>流量分配不均衡。包含但不限于多台机器计算机资源，包含但不限于CPU资源，内存资源。经常导致cpu彪高，jvm gc频繁。<br/>接口的容错性不高，感知问题的实时性不高，以及数据补偿不够及时切灵活性不高。<br/>整体系统的配置化能力，后台管理能力，自动化能力较弱。<br/>这些问题导致了系统应对大量数据和突发流量时，表现较弱，需要经常人工干预，占用大量人力资源。<br/>对流量和以及异常数据的自动化控制较弱。<br/>需要解决: 灵活性能力，监控能力，自动化感知和自动化数据补偿能力。<br/>数据错误原因: 三方店铺授权失效，三方接口限流，三方接口报错<br/>技术问题:  定时任务的颗粒度较大。 分布式任务负载不均衡。数据补偿需要人工干预。造成系统资源浪费， 三方接口流量浪费。<br/><br/>关键的改造点: <br/>任务颗粒度降低<br/>任务节点分配按照主播等级和直播时间自动化资源分配 （让高优先级的主播使用较高配置的机器资源）<br/>任务自动挂起，让出机器资源 （有的任务可以停掉，让出机器资源以及三方接口流量使用额度）<br/>任务的执行频率自动化 （数据较少时候降低数据拉去频率）<br/>子任务拉去数据的时间参数步长自动化调整 (数据较少时候，减少请求三方接口请求次数)<br/>子任务请求时间参数拆分自动化 （数据流量较高时，提高数据拉去实时性）<br/>任务补偿自动化（接口异常时，后台任务自动化补偿数据）<br/>日志记录完整性 （记录错误接口类型，执行错误的参数，便于下次重试）<br/>什么情况进行任务频率的调整，什么情况挂起任务， 什么时间调整任务参数分片 ，什么时间调整单台机器的线程数量<br/><br/><br/><br/>系统优化后的技术架构: <br/>完整配置: 主播最新直播日期， 主播近期合作店铺<br/>xxl job ，数据拉去任务， 异常数据<br/>mysql 分库分表<br/>redis 缓存配置信息，主播信息，失败任务信息<br/>rocketmq 详情接口请求流量削峰<br/>elastisearch 提供数据查询能力<br/>sentinel 限流能力<br/>granfana 系统资源，流量监控<br/><br/><br/><br/> | | |

| # 盖亚客服系统<br/>该系统整体模块为工单中心，数据中心，售后中心，售后登记，物流监控，返款管理。盖亚客服系统是一个服务每一位在直播间下单的辛选用户，通过线上和线下结合的方式保证数据的线上化。提供核心业务处理能力包含但不限于工单中心 (保证客服人员的处理效率) ，数据中心 (辅助客服人员定位业务信息) ，直播返款（提供个性化的优惠力度保证辛选用户的用户粘性）<br/><br/>工单中心: <br/>提供工单处理能力，包括工单分配，工单跟进，工单状态流转。<br/>使用 mysql 作为数据存储，redis 提供缓存数据能力，elasticsearch 提供工单查询和分析能力。<br/>提供数据分析能力，开发数据分析模块: 分析工单系统数据，包含数据统计，数据预警，团队排行。<br/>工单数据量达到3000万+ ，通过优化mysql 索引提高接口响应的RT，通过elasticsearch 搜索提高数据检索能力，和数据的聚合分析能力。<br/>使用 rocketmq 进行工单处理长链路的解耦操作。<br/>使用定时任务XXL job 对业务数据聚合分析，生成中间态的统计数据。<br/>采用分布式锁，保证工单在流转过程中得状态数据一致性。<br/><br/>数据中心: <br/>提供基础业务数据的查询能力，辅助客服人员进行相应的业务处理，如商品查询，订单查询，直播记录查询，物流查询。<br/>数据主要依赖数据同步平台，提供基于elasticsearch redis 高性能查询接口。<br/><br/>直播返款:<br/>提供特色的直播活动优惠，管理直播过程中主播承诺的商品返现能力。<br/>包含商品返款计划的管理，返款订单的管理，返款信息的管理，返款申诉管理。通过线上和线下返款相结合的方式，很好的服务用户。<br/>返款链路过长，使用rocketmq 进行业务解耦操作，通过消息事务的方式保证接口最终一致性。<br/><br/>个人核心职责:  <br/>核心业务模块的系统开发。<br/>利用多线程技术，优化业务数据模型，索引提高接口的相应速度。<br/>通过系统调优减轻流量过大照成的服务器压力，包括 JVM调优，MySQL索引调优。<br/>利用各种中间件管理管理系统海量数据，保证数据的安全性和完整性。<br/><br/>系统技术框架:<br/>采用分布式微服务的系统架构。<br/>微服务能力: SpringCloudAlibaba，Dubbo <br/>系统基础能力: Spring SpringBoot Mybatis，多线程工具， CompletableFutureTask<br/>数据存储: Mysql , Redis，Elasticsearch<br/>消息中间件: RocketMq<br/><br/><br/><br/><br/><br/><br/><br/><br/> | | |

| 例如:城市公共设施维修：该模块旨在通过智慧化手段提高城市公共设施维修的效率，包括设施巡检、故障上报和维修流程。具体业务流程包括下面如下处理逻辑设施巡检：使用智能设备对公共设施进行巡检，提高巡检效率和质量。<br/>故障上报：市民可以通过手机应用等方式进行设施故障上报，提高故障上报的效率和准确率。 <br/>维修流程：通过智能派单和智能维修工具等手段，实现对设施故障的快速响应和处理，提高维修效率和质量。  <br/><br/> 1、参与代码结构重构：利用设计模式适配器 对service 接口层代码进行了重构。将用户的业务行为抽象为代码中的技术行为 saver updater selecterdeleter分组管理，统一维护。对于业务开发人员来说可以很清晰的看到一个核心模块中的业务方法以及使用的业务场景，达到快速熟悉和开发具体业务的目的。对于项目管理者可以清晰的看到 方法的命名以及对不同开发人员开发规范和命名规范。降低系统的代码的维护成本以及提供代码的可读性。<br/> 2、数据对接开发: 利用模版方法模式完成第三方数据的同步。编排好业务的处理逻辑 相同的逻辑放在抽象方法的具体实现中，不同的逻辑交给具体的实现类进行处理。 用轮训的方式请求接口，将每次获取数据的标志位放入数据库统一管理。用分页和时间排序的方式获取数据并导入业务系统。<br/>3、数据一致性优化开发: 数据同步的改进， 之前是手动设置通过硬编码进行实现， 手动将mysql更新好的数据推送到es和缓存redis中，利用框架对事务的管理发送消息， 后期采用了 canal组件 进行binlog的订阅将消息推动到mq然后推送到es和缓存中。核心业务数据字段用定时任务对数据检索一致性进行兜底。保证最终一致性。<br/>4、业务接口优化开发：统计业务数据的查询优化，开发背景：统计数据维度过多，客户要求实时行无法采用缓存。接口中有多次对数据库的聚合查询操，因为是串行行查询，个别表的数据过多影响整体的性能。使用异步查询。利用completeablefuturetask这个工具对线程进行异步编排最终将数据放入vo中响应给客户端。降低接口的rt。<br/> 5、业务中的核心数据例如流程中事件信息在系统中的查询优化开发，利用搜索引擎es提高数据的检索效率，使用canal组件保证数据一致性。<br/>6、 流程节点中流程走向业务开发，以及流程设计，比如系统中核心流程：上报->立案->分拨->处置->结案->流程处理评价。 <br/>7、 根据产品提供的产品文档设计相关模块的技术实现。  <br/><br/>基础框架:SpringBoot、SpringCloudAlibaba、Feign、Nacos、GateWay、Seata<br/>数据库:Mysql <br/>缓存中间件:Redis <br/>消息中间:RocketMQ <br/>数据搜索引擎:Elasticsearch<br/> 数据同步组件：Canal<br/> 流程引擎管理: bpm流程引擎Activiti7<br/>项目结构: 面向Starter 结构进行业务逻辑的插拔扩展和技术组件的插件扩展。满足不同局点可能会提出自己的定制化接口和业务逻辑。  <br/><br/><br/><br/><br/> | | |
