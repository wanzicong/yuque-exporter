

:::info
产品设计->技术方案设计->测试用例设计->编码开发->单元测试->业务测试->实际交付

整体提效思路: 

1. 文档标准化，产品设计基于 prompt 模板设计产品思路，业务逻辑描述模板化，目的是为了让 AI 基于业务逻辑做出后续的生成
2. 通过提示提示词语 prompt 描述，基于上游产品输出的产品文档，让 AI 帮助生成对应的技术方案设计文档，比如可以告诉现有的代码结构，使用技术栈，包括一些定制化的业务描述。
3. 基于产品文档，技术方案文档，加上简单的定制化提示词，生成对应的测试用例，可以是 md 格式的，可以是思维导图形式的
4. 通过 AI 编码工具进行，简单代码业务结构设计，代码提示，代码注释生成，代码潜在问题发现，代码性能优化
5. 根据每个方法的代码 + 业务逻辑的描述产出服务需求的单元测试用例，用来覆盖 70% 的单链路测试业务代码生成

备注: 

1. 上面的每个环节相互配合，从而整体提高团队协助效率
2. 因为单个环节很难做到效率的大幅度提高，只有每个环节相扣互相配合才能得到更大的产出比

**所有提效的模式就是要利用好提示词！！！！！**

:::



# 产品设计(基础)
产品设计文档标准化，模板化，为下游环节更好的合作提效做出坚实的基础

<font style="color:#DF2A3F;">提效建议: 将非结构化的需求描述，设计，背景，功能点，多业务模块联动，交给 AI 输出相对结构化的文档。</font>

要求: 需求设计还是采用图文结合的方式比较好，因为 AI 目前对文字的理解更好

具体模板需要产品内部讨论

大致方向如下:

1. 需求名称
2. 需求背景
3. 需求相关模块，多模块，模块之间依赖
4. 需求影响范围，相关页面
5. 需求细节
    1. 功能点描述
    2. 功能点要求 字段相关，交互相关，逻辑相关
    3. 功能要求，如性能要求，该功能与另外功能的相互作用
    4. 业务逻辑，包含核心逻辑，边界逻辑等



AI 参与环节（通用大语言模型）：需求结构化，需求逻辑建议，需求逻辑补充



# 技术方案设计(开发准备阶段)
<font style="color:#DF2A3F;">AI 提效: 帮助开发人员理解需求，整理需求，技术方案框架输出，开发人员基于现有的结构进行补充完善~</font>

依赖需求文档理解，将需求文档(文字版)+通用大语言模型+提示词语+开发人员的额外要求=结构化的技术方案

提示词语:  

你是某某方面专家

请更具具体的需求文档设计出一份可行性高的技术方案文档

当前使用的技术框架是(Spring + Mysql + ....)

方案包含具体的业务逻辑框架，业务流程图，接口文档，表结构设计,缓存设计，消息设计....



使用方式: 

利用大预言模型进行对话

内容 具体需求文档+prompt 提示词



# 测试用例设计(质量保证)
## 开发前
1. 基于产品文档，技术方案文档 进行用例生成。 可以基于测试用例方向进行补充，减少测试人员的投入时间。
2. 使用通用大预言模型，可以进行不同维度的用例输出，比如正常测试，异常测试，暴力测试，性能测试，随机测试
3. 依赖描述描述好的提示词模板 + 文档 + 手动补充 = 完整的测试用例。

- Role: 软件测试专家
- Background: 用户需要编写测试用例，以确保软件在各种环境下都能稳定运行，满足业务需求。测试用例需要全面覆盖环境问题、配置问题和业务问题，确保软件的质量和可靠性。
- Profile: 你是一位经验丰富的软件测试专家，精通各种测试方法和工具，对软件测试的各个环节有着深刻的理解和丰富的实践经验，能够从多个角度发现潜在问题。
- Skills: 你具备全面的测试技能，包括但不限于环境搭建、配置管理、业务流程分析、测试用例设计、自动化测试脚本编写以及缺陷跟踪与管理。
- Goals: 设计一套全面的测试用例，覆盖环境问题、配置问题和业务问题，确保软件在不同场景下都能正常运行，满足用户需求。
- Constrains: 测试用例应简洁明了，易于理解和执行，同时确保测试的全面性和有效性。测试用例应基于实际业务场景，避免冗余和重复。
- OutputFormat: 文字描述、表格形式、流程图辅助说明。
- Workflow:
  1. 分析软件需求和业务流程，明确测试目标和范围。
  2. 根据需求设计测试用例，覆盖环境问题、配置问题和业务问题。
  3. 执行测试用例，记录测试结果，跟踪和管理缺陷。
- Examples:
  - 例子1：环境问题测试用例
    - 测试目标：验证软件在不同操作系统下的兼容性。
    - 测试步骤：
      1. 在Windows 10、Windows 11、macOS Monterey、macOS Ventura、Ubuntu 20.04、Ubuntu 22.04等操作系统上安装软件。
      2. 启动软件，检查是否能正常加载界面。
      3. 执行主要功能操作，如文件打开、保存、数据查询等，检查是否存在异常。
    - 预期结果：软件在所有测试操作系统上都能正常启动并执行主要功能。
    - 实际结果：记录软件在每个操作系统上的表现，包括启动时间、功能执行情况等。
    - 缺陷记录：记录发现的任何兼容性问题，如界面显示异常、功能无法使用等。
  - 例子2：配置问题测试用例
    - 测试目标：验证软件在不同网络配置下的性能。
    - 测试步骤：
      1. 配置不同的网络环境，包括高速宽带（100Mbps）、中速宽带（20Mbps）、低速宽带（5Mbps）、4G网络、5G网络等。
      2. 在每种网络环境下启动软件，测量软件的加载时间。
      3. 执行数据同步操作，记录同步时间、同步成功率等指标。
    - 预期结果：软件在所有网络环境下都能正常加载，数据同步成功率应达到95%以上。
    - 实际结果：记录软件在每种网络环境下的加载时间和数据同步情况。
    - 缺陷记录：记录发现的任何性能问题，如加载时间过长、数据同步失败等。
  - 例子3：业务问题测试用例
    - 测试目标：验证软件的订单处理功能是否符合业务需求。
    - 测试步骤：
      1. 创建一个新订单，输入客户信息、产品信息和订单金额。
      2. 提交订单，检查订单是否成功生成并保存到数据库。
      3. 检查订单状态是否正确更新为“待处理”。
      4. 执行订单处理操作，将订单状态更新为“已发货”。
      5. 检查订单状态是否正确更新，并通知客户。
    - 预期结果：订单处理流程符合业务逻辑，订单状态更新正确，客户通知及时。
    - 实际结果：记录订单处理的每一步操作结果，包括订单状态、客户通知情况等。
    - 缺陷记录：记录发现的任何业务逻辑问题，如订单状态更新错误、客户通知失败等。
- Initialization: 在第一次对话中，请直接输出以下：您好！作为专业的软件测试专家，我将协助您编写全面的测试用例。请告诉我您要测试的软件的具体需求和功能，以及您希望重点关注的测试领域。

## 开发环节 AI 可以帮助哪些？
<font style="color:#DF2A3F;">是否前置流程进行了标准化？</font>

1. 文档标准化了（包含产品方案文档，技术方案文档）

 如果文档进行了模板化，可以通过提示词工程+代码工程结构描述，生成对应的业务逻辑框架

 包含但不限于，方法定义，类定义，表结构定义，性能考虑方案等等

2. 不依赖文档进行提效
    1. 代码提示自动补全
    2. 工具代码生成
    3. 单个方法注释生成
    4. 代码关键 log 生成
    5. 代码性能逻辑优化
    6. 代码潜在问题发现并修复

## 工具推荐
推荐工具： tongyilingma ， trace 

自建标准化 AI 流程: 代码结构+批量操作（包括批量单元测试生成，批量 bug 发现，批量代码优化建议生成......）

## 使用分析
通用工具的局限: 只能按照现有功能单个操作，无法做到批量化流程化

推荐自定义工具: api 对接 + 自定义代码解析+ 自定义输出形式



## 使用建议
开发过程中包含了哪些具体的内容？

1. 基础逻辑架构设计（AI 可帮助）
2. 基础工具方法封装 (AI 可帮助)
3. 具体需求实现 (过度依赖人工)

如果业务总包含了 100 行代码，目前 AI 通过提示工程可做到基础重复的代码生成，占比可到达 10% -15%



## 未来期望
1. 通过每个环节的标准化或者单个环节的标准化、模板化加上 AI 辅助业务代码生成占比达到 20%-25%
2. 建立需求+代码的 rag 知识库能力，比如通过问答的方式可让不熟悉业务的人快熟熟悉业务系统
3. 建立代码 AI 审查机制，问题提前发现机制，报警机制，分析结果报表导出能力
