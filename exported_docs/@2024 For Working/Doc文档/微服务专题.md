## 配置中心) 
###  为什么要用配置中心
统计化管理每个集群的配置， 微服务架构中 会有非常多的服务， 如果每个服务都使用自己的配置管理， 

管理成本很大，而且容易出错。 使用配置中心可以简化配置的管理， 可以灵活的切换服务的环境

### SpringCloud 可以选择哪些配置中心
springcloud 中 可以使用 spring cloud config ，nacos ， 阿波罗作为配置中心

### Nacos 配置中心原理了解过吗
配置中心的原理主要提供配置的集中管理， 可以将配置信息持久化到数据库中， 

作为配置中心最核心的功能就是， 当配置发生配置变更的时候服务能动态感知的配置的变化

同时在应用程序中能使用最新的配置对外提供服务， nacos 实现配置动态感知的方案是采用

长轮训的方式。

### Nacos配置中心长轮序机制
客户端会定时向server 端发送获取最新配置的请求， 当server端接受到请求后，会检查当前配置有没有发生变化， 

如果没有发生变化，server端不会立即返回，而是保持当前连接，监听当前的配置变化， 如果有变化就立即返回，

没有变化就一直等待知道链接断开， 链接断开后client 端会继续发送请求， 获取最新配置





## 远程调用）
### 能说下HTTP和RPC的区别
http 和 rpc 都可以用来远程服务调用， http 是web 传输协议， rpc 是一种远程服务调用协议。

通过本地方法的调用触发远程服务方法的执行。 不同的框架实现的规则不同，基本都是对rpc 协议的封装， 采用不同的序列化算法

常见的rpc 框架有 dubbo grpc 

两者并不互斥， dubbo 也可以使用 http 作为通讯协议

### Feign和Dubbo的区别
feign 和 dubbo 都可以用来远程服务调用

feign 远程服务调用 底层依赖的是 http 超文本传输协议。通过对resttemplate 的封装。

通过对接口的代理，来实现具体的调用细节。

feign 底层采用的是 ribbo 进行服务的负载均衡。



dubbo 是一款高性能的远程服务调用框架， 是阿里巴巴开源的rpc 框架， 目前已经交给Apache 来维护

dubbo 底层依赖是 rpc 通讯协议， 数据层采用各种序列化协议， 比如json 序列化 ，二进制序列化， protocol buffer 序列化协议等

负载均衡通过框架本身提供的算法



服务之间的注册与发现， feign 主要是依赖spring cloud 的微服务解决方案， 如 常用的 euraka 和 nacos

dubbo 服务注册与发现 大多依赖的是 zookeeper



### 说下Feign吧
feign 是一个rpc 调用的组件 ，也可以说是一个远程服务调用的组件，

提供负载均衡 和 服务调用的功能，核心功能是对 resttemplate 和 ribbo 的分装

通过代理的方式， 根据服务名称获取服务列表 ， 通过负载均衡进行远程服务调用

通过封装简化 远程服务调用，使远程服务调用看起来就像是调用本地方法一样

### 为什么-feign-第一次调用耗时很长
因为服务注册列表是通过懒加载的形式， 当服务第一次调用时候，系统中没有缓存具体的服务列表信息， 

需要请求注册中心，获取对应需要调用的服务列表，然后将获取到的服务信息放置到缓存当中。

当下次请求的时候优先从缓存中获取服务信息， 当服务发送变化的时候，拉去最新的服务列表，刷新缓存。

### Feign如何实现认证传递
feign 本身是依赖http 传输协议， 可以将认证信息放在请求头中，通过拦截器解析请求头， 将认证信息传递给下游服务



### Feign如何做负载均衡
feign 是通过 resttemplate 进行调用服务， 通过负载均衡组件 ribbo 进行发现服务

通过 服务名称获取服务列表， 通过负载均衡算法定位到具体需要调用服务基本信息，

进行服务的调用。



### 说一下都有那些负载均衡算法
常见的负载均衡算法： 

轮训算法， 随机， 一致性hash ，权重算法，随机权重算法

## 服务容灾）
### 什么是服务雪崩
当一个服务或者多个服务不可用的时候， 上游服务持续发送请求， 

导致服务阻塞大量请求，影响整体链路服务的性能，从而导致全部服务的负载过高，最终导致服务的不可用

### 什么是服务熔断、什么是服务降级
当服务达到某个限制后，比如单位时间内的失败次数，相应时间过长等等， 

会触发服务的熔断，从而返回默认数据，或者错误页面，防止大量请求堆积， 影响上下游链路的调用



服务降级： 当系统整体流量过高，为了保证系统中核心业务不会受到影响，让部分服务对外服务时返回默认数据，或者缓存数据，

从而达到保护整体系统的目的

###  hystrix-怎么实现服务容错
hystrix 提供服务的熔断，降级，请求缓存，请求合并，线程池隔离，以及实时监控确保服务的高可用

### sentinel-怎么实现限流的
sentinel 提供对资源监控的方式， 该资源可以是url 也可以是业务方法， 对资源的请求进行上报和监控，和配置的限流策略比较，来控制服务时候限流，

以及限流的模式是什么样的， 比如令牌桶限流，漏斗限流

网关是一个代理服务， 可以让前端只需要具体的服务名，就可以调用到具体的服务节点

简化了前端与服务端的交互，

网关通过路由转发的功能，把请求发送的具体的微服务

提供了以下功能:  路由转发， 负载均衡，请求拦截等操作

### springcloud-可以选择哪些-api-网关
spring cloud gateway

zuul 

### spring-cloud-gateway-核心概念
路由， 断言， 过滤器

路由：将请求定位到具体的服务

断言： 定义拦截规则匹配

过滤器： 执行断言定位到的具体请求的拦截操作

## 分布式事务
[https://javabetter.cn/sidebar/sanfene/weifuwu.html#%E5%88%86%E5%B8%83%E5%BC%8F%E4%BA%8B%E5%8A%A1](https://javabetter.cn/sidebar/sanfene/weifuwu.html#%E5%88%86%E5%B8%83%E5%BC%8F%E4%BA%8B%E5%8A%A1)

### seata-支持哪些模式的分布式事务
分布式事务，是将多个本地事务组合在一起，形成一个完整的分布式形式的事务，

分布式事务，一般会分为， 强一致性事务，和最终一致性事务



Q: 分布式有哪些解决方案

A： 2pc 3pc tcc saga 分布式消息事务

<font style="color:rgb(47, 50, 56);">2pc 是一种机制</font>

<font style="color:rgb(47, 50, 56);">依靠数据库实现分布式事务</font>

<font style="color:rgb(47, 50, 56);">强一致性， 全局阻塞</font>

<font style="color:rgb(47, 50, 56);">事务参与者 事务协调者</font>

<font style="color:rgb(47, 50, 56);">第一个阶段 ， 事务协调者通知事务参与者，事务进行全局阻塞，当前已经开启事务， 还没提交，保证事务的隔离性，采用serializable 的隔离级别 可能的问题，如果出现某个事务的网络问题，其他事务全局进行阻塞，浪费性能。</font>

<font style="color:rgb(47, 50, 56);">第二阶段: 通知提交或者回滚</font>

<font style="color:rgb(47, 50, 56);"></font>

<font style="color:rgb(47, 50, 56);">3pc 在2pc 的基础上增加了询问阶段，防止网络问题导致的全局事务阻塞，这个阶段事务参与者不会进行任务事务操作</font>



为了减少阻塞力度 出现了TCC

类似于2pc ，更加依赖代码逻辑 Try Confirm Cancel

灵活性更好了， 使用代码逻辑排除造成事务问题的可能性，提前准备好全部的事务的准备工作



有的事务是链路比较长 可以使用saga 

适合长事务， 隔离性没有要求

使用反向操作的方式， 比如正常情况创建订单，异常情况就删除订单



使用消息事务， 保证最终一致性

1. 先发送预备消息，也就是半消息，放在事务消息队列中，当前消息不可消费
2. 开启本地事务
3. 事务成功， 提交本地事务，设置消息可消费
4. 消息会定期询问本地事务情况

使用消息完成事务， 假定业务代码是没有问题的



总结: 2pc 3pc 都是强一致性， 事务全局阻塞

tcc saga 消息事务 都是最终一致性，通过重试，业务兜底，人工补偿的方式保证最终一致性

Q: 该如何选seata 事务类型

A:  tcc saga

at ，xa

AT 模式是seata 中独有的事务模式 （默认事务）

通过生成反向sql回滚操作，在unlog 表中

使用AP 模式，允许出现短时间的不一致情况，保证数据的最终一致

通过对账和业务兜底以及人工补偿的机制来保证最终一致性

  
XA： 是数据库中的一种分布式事务协议，保证系统的2pc 事务 ，是2PC 理论的落地

依赖数据的事务， CP模式，保证强一致性

使用场景一般是对数据敏感的，不接受出现数据不一致的场景， 比如金融行业
